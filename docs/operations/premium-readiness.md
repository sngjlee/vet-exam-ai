# 결제/프리미엄 준비 체크리스트

본 문서는 KVLE에 결제, 구독, 프리미엄 기능을 도입하기 전 반드시 확정해야 하는 내부 기준입니다.
현재 원칙은 **결제 구현을 서두르지 않고, 법무/운영/상품 기준이 준비된 뒤에만 결제 플로우를 붙인다**는 것입니다.

본 문서는 법률 자문이 아니며, 공개 약관 또는 결제 화면에 반영하기 전 별도 검토가 필요합니다.

## 1. 도입 원칙

- 무료 핵심 학습 흐름은 유지합니다.
- 유료 기능은 시험 대비 효율을 높이는 보조 기능으로 둡니다.
- 결제 전환은 학습 데이터를 인질로 삼지 않습니다.
- 결제 실패, 환불, 해지 상태에서도 기존 무료 데이터 접근을 불필요하게 막지 않습니다.
- 운영자가 직접 설명할 수 없는 과금 구조는 도입하지 않습니다.
- Stripe 또는 기타 결제 연동은 아래 선행 조건이 완료되기 전까지 보류합니다.

## 2. 선행 조건

| 영역 | 완료 기준 | 상태 |
|---|---|---|
| 사업자 정보 | 사업자등록 여부, 공개 상호, 대표자, 주소, 연락처 확정 | 대기 |
| 통신판매 | 통신판매업신고 필요 여부 및 신고 정보 확정 | 대기 |
| 환불 정책 | 무료 체험, 부분 환불, 기간제 상품, 디지털 콘텐츠 청약철회 제한 기준 확정 | 대기 |
| 약관 | 이용약관에 가격, 결제, 해지, 환불, 자동결제 조항 반영 | 대기 |
| 개인정보 | 결제 대행사, 영수증, 결제 식별자, 보관 기간, 처리위탁 반영 | 대기 |
| 고객지원 | 결제 문의, 환불 요청, 영수증 문의 접수 채널 확정 | 대기 |
| 회계 | 매출 확인, 환불 기록, 정산 확인 루틴 확정 | 대기 |
| 운영 권한 | 관리자 화면에서 유료 상태를 임의 조작할 수 있는 범위와 감사 로그 확정 | 대기 |

## 3. 프리미엄 후보 기능

초기 유료 기능은 새 콘텐츠를 잠그기보다, 이미 있는 학습 흐름을 더 정교하게 만드는 기능이 적합합니다.

| 후보 | 무료 유지 범위 | 프리미엄 후보 범위 | 우선순위 |
|---|---|---|---|
| 미니 모의고사 | 기본 20문제 모의고사, 최근 로컬 히스토리 | 장기 히스토리, 과목별 추세, 회차별 리포트 | 높음 |
| 약점 분석 | 기본 정답률, 최약 과목 | topic/tag 단위 약점 리포트, 추천 학습 순서 | 높음 |
| AI 해설 보조 | 공식 해설, 댓글 노하우 | 질문형 해설 보조, 오답 원인 설명 | 중간 |
| 시험 직전 플랜 | D-day, 오늘 할 일 | 남은 기간 기반 주간 플랜, 모의고사 기반 보정 | 중간 |
| 검색 고도화 | 기본 문제/해설/댓글 검색 | 저장 검색, 유사 문항 묶음, 개념별 모아보기 | 낮음 |

도입 전 각 기능은 “무료 사용자가 서비스 가치를 이해하는 데 필요한가”와 “유료 사용자가 반복적으로 비용을 낼 만큼 자주 쓰는가”를 분리해서 판단합니다.

## 4. 과금 모델 후보

| 모델 | 장점 | 위험 |
|---|---|---|
| 월 구독 | 구현과 안내가 단순하고 기능 실험이 쉽다 | 시험 직전 외 기간의 유지율이 낮을 수 있다 |
| 6개월 패스 | 시험 준비 기간과 맞고 환불 계산이 비교적 명확하다 | 중도 환불 정책을 먼저 정해야 한다 |
| 12개월 패스 | 장기 수험생에게 안정적이다 | 초기 신뢰가 낮으면 전환이 어렵다 |
| 후원형 | 결제 압박이 낮고 커뮤니티 정체성과 맞다 | 안정적 매출 예측이 어렵다 |

초기 권장안은 **무료 핵심 + 6개월 패스 실험**입니다. 다만 실제 가격과 기간은 베타 사용자의 반복 사용 데이터가 쌓인 뒤 결정합니다.

## 5. 결제 구현 전 기술 체크

- `profiles` 또는 별도 entitlement 테이블에 유료 상태를 저장할지 결정합니다.
- 결제 대행사의 customer id, subscription id, payment id를 어떤 테이블에 저장할지 정합니다.
- 결제 원장성 이벤트는 덮어쓰기보다 append-only 로그로 저장합니다.
- webhook idempotency key를 저장해 중복 이벤트를 막습니다.
- 관리자 수동 부여/회수는 `admin_audit_logs`에 남깁니다.
- 유료 권한 체크는 클라이언트 표시가 아니라 서버/RLS 기준으로 보호합니다.
- 결제 실패, 구독 해지, 환불, chargeback 상태를 모두 별도 상태로 모델링합니다.
- Preview/Staging에서는 실제 결제와 테스트 결제 키가 절대 섞이지 않게 합니다.

## 6. 공개 화면 체크

유료 기능을 켜기 전 아래 화면이 먼저 준비되어야 합니다.

- 가격표 또는 유료 기능 안내
- 결제 전 최종 확인 화면
- 결제 완료 화면
- 결제 실패 화면
- 구독/패스 상태 확인 화면
- 해지 또는 환불 안내 화면
- 영수증/문의 안내
- 유료 기능 접근 제한 상태
- 약관, 개인정보 처리방침, 환불 정책 링크

## 7. 운영 체크

매일:
- 결제 실패 이벤트 확인
- 환불/문의 접수 확인
- webhook 오류 확인

매주:
- 매출, 환불, 활성 유료 사용자 수 확인
- Sentry 결제 관련 오류 확인
- 관리자 수동 권한 변경 audit 확인

매월:
- 정산 내역 확인
- 환불 처리 표본 점검
- 약관/가격/기능 안내와 실제 서비스 동작 일치 여부 확인

## 8. 출시 게이트

결제 기능은 아래 조건을 모두 만족할 때만 production에 켭니다.

- 사업자/통신판매/환불/약관/개인정보 항목이 확정되었습니다.
- 결제 테스트 키와 운영 키가 환경별로 분리되었습니다.
- webhook 재시도와 중복 이벤트 처리가 검증되었습니다.
- 결제 성공, 실패, 해지, 환불, 관리자 수동 부여/회수 smoke test가 통과했습니다.
- 유료 권한이 없는 사용자가 프리미엄 API 또는 데이터에 접근할 수 없습니다.
- 환불 또는 해지 후 무료 핵심 기능 접근이 정상입니다.
- 결제 장애 시 유료 기능을 일시적으로 열거나 닫는 운영 기준이 정해졌습니다.

## 9. 보류 항목

아래 작업은 지금 하지 않습니다.

- Stripe Checkout 또는 Billing 구현
- 카드 결제 UI 구현
- 자동결제 활성화
- 프리미엄 권한 테이블 migration
- 공개 가격표 게시
- 유료 기능을 전제로 한 약관 시행

다음 단계는 실제 결제 구현이 아니라, 베타 사용자의 미니 모의고사/약점 분석 사용 데이터를 보고 유료 후보 기능의 사용 빈도와 가치를 확인하는 것입니다.
