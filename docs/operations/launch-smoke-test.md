# 런칭 최종 QA / 스모크 테스트

본 문서는 소프트 런칭, 정식 공개, 큰 기능 배포 직후 실행하는 수동 QA 기준입니다.
`docs/operations/production-readiness-checklist.md`가 환경변수와 운영 설정을 확인한다면, 이 문서는 실제 사용자 흐름이 깨지지 않았는지 짧게 확인합니다.

비밀번호, magic link, 가입 증빙 원본 path, service role key, `CRON_SECRET`은 테스트 기록이나 스크린샷에 남기지 않습니다.

## 1. 실행 시점

- 소프트 런칭 직전 staging 또는 production preview에서 1회 실행합니다.
- 정식 공개 직전 production에서 1회 실행합니다.
- 인증, RLS, 댓글, 검색, 이미지, 관리자, 결제 관련 변경 배포 직후 해당 섹션을 다시 실행합니다.
- DB migration, Storage policy, Vercel env, Supabase key 변경 배포는 전체 smoke를 실행합니다.

## 2. 준비물

| 계정/권한 | 목적 | 준비 기준 |
|---|---|---|
| 비로그인 브라우저 | 공개 페이지, robots, 로그인 전 UX 확인 | 시크릿 창 또는 로그아웃 세션 |
| 승인 대기 계정 | 가입 증빙 제출/대기 화면 확인 | `signup_status=pending` |
| 승인 회원 | 학습, 댓글, 신고, 설정 확인 | 일반 사용자 role, active 상태 |
| 관리자 | admin 화면, 승인/신고/감사 로그 확인 | `role=admin`, active 상태 |
| 테스트 문항 | 댓글/오답/검색/이미지 확인 | 공개 가능한 대표 문항 2~3개 |

테스트 계정 이메일과 증빙 파일은 운영자만 접근 가능한 별도 안전한 위치에서 관리합니다. 문서나 이슈에는 마스킹한 식별자만 남깁니다.

## 3. 통과/중단 기준

### 즉시 런칭 중단

- 로그인/회원가입/가입 증빙 제출이 500 또는 무한 로딩에 빠집니다.
- 승인 회원이 문제 풀이, 문제 상세, 검색, 복습 중 하나도 사용할 수 없습니다.
- 비승인 계정이 승인 회원 전용 화면이나 board/admin 기능에 접근합니다.
- 일반 회원이 관리자 화면 또는 관리자 mutation을 실행할 수 있습니다.
- 댓글/신고/이미지 업로드에서 다른 사용자의 비공개 데이터가 노출됩니다.
- `/admin/ops`에서 필수 env 누락, cron secret 누락, service role 누락이 보입니다.
- Sentry에 민감 원문이 포함된 오류 이벤트가 들어갑니다.

### 런칭 가능하나 이슈 등록

- 일부 empty state 문구가 어색하지만 기능은 동작합니다.
- 댓글 수, 알림 수, 통계 값이 지연 갱신되지만 새로고침 후 정상입니다.
- 특정 브라우저에서 비핵심 hover/animation만 깨집니다.
- 운영자 보조 화면의 필터/정렬이 일부 어색하지만 원본 데이터 확인이 가능합니다.

## 4. 공개/비로그인 흐름

| 확인 | 기대 결과 |
|---|---|
| `/` | 랜딩이 200으로 열리고 주요 CTA가 로그인/회원가입으로 이동합니다. |
| `/guide` | 처음 이용 가이드가 열리고 상단 메뉴를 늘리지 않아도 주요 기능으로 이동할 수 있습니다. |
| `/terms`, `/privacy`, `/community-guidelines` | 사용자 공개 정책 문서가 최신 사본으로 렌더링됩니다. |
| `/questions`, `/search`, `/quiz`, `/comments` | 공개 허용 범위에 맞게 열리거나 로그인 유도/제한이 일관됩니다. |
| `/admin` | 비로그인 사용자는 로그인 또는 권한 제한 흐름으로 이동합니다. |
| `/robots.txt` | 공개 상태와 베타/noindex 상태가 `NEXT_PUBLIC_INDEXING_ENABLED`와 일치합니다. |

## 5. 가입/인증 흐름

| 확인 | 기대 결과 |
|---|---|
| 로그인 페이지 | 이메일/비밀번호 입력, 오류 문구, 가입 전환이 정상입니다. |
| 신규 가입 | 가입 후 승인 대기 또는 증빙 제출 화면으로 이동합니다. |
| 가입 증빙 제출 | 허용 파일만 업로드되고 성공 후 pending 상태가 표시됩니다. |
| 승인 전 접근 | dashboard/board/settings 등 승인 필요 화면은 `pending-proof` 또는 `pending-review`로 제한됩니다. |
| 비밀번호 재설정 | reset 링크 발급/입력 흐름이 동작하고 reset token이 로그나 화면에 남지 않습니다. |
| 로그아웃 | 세션이 끊기고 인증 필요 화면 접근이 차단됩니다. |

## 8. 알림/프로필/계정 흐름

| 확인 | 기대 결과 |
|---|---|
| 알림 벨 | 읽지 않은 알림 수와 드롭다운이 정상이며 읽음 처리 후 수가 줄어듭니다. |
| 프로필 | 닉네임 설정/프로필 조회/내 댓글 목록이 권한에 맞게 보입니다. |

| 계정 삭제 | 확인 절차 없이는 실행되지 않고, 실행 후 재로그인/데이터 접근이 제한됩니다. |

## 9. 관리자 흐름

| 확인 | 기대 결과 |
|---|---|
| `/admin/ops` | env, site URL, robots, service role, cron 최근 실행 상태가 표시됩니다. |
| `/admin/sentry-test` | server/client test event가 Sentry에 도착하고 tag 기준이 맞습니다. |
| 가입 신청 | 승인/거절 처리 후 감사 로그가 남고 증빙 보존/삭제 기준과 일치합니다. |
| 신고 처리 | 댓글/건의 신고를 처리하면 공개 노출 상태와 audit log가 일치합니다. |
| 사용자 관리 | 역할 변경, 정지, 비밀번호 재설정 링크 발급이 audit에 남습니다. |
| 문제 관리 | 문제 수정, 이미지 트리아지, 품질 대시보드 진입이 500 없이 동작합니다. |
| 감사 로그 | 최근 관리자 액션을 action/resource 기준으로 검색할 수 있습니다. |

## 10. 배포 후 운영 확인

1. `npm run check:migrations`, `npm run lint`, `npm run typecheck`, `npm run build`가 통과한 커밋인지 확인합니다.
2. Vercel production deployment가 성공했고 대상 commit SHA가 의도한 값인지 확인합니다.
3. `/admin/ops`에서 fail이 0인지 확인합니다.
4. `/admin/sentry-test`로 server/client 이벤트를 보내고 Sentry에서 `area`, `operation`, `source` tag를 확인합니다.
5. Vercel Cron 로그와 `/admin/ops` 최근 cron 실행 표가 일치하는지 확인합니다.
6. Supabase Auth, Database, Storage 대시보드에 예상치 못한 오류 spike가 없는지 확인합니다.
7. Sentry 새 이슈가 있다면 사용자 영향, 재현 경로, 민감정보 포함 여부를 먼저 분류합니다.

## 11. 결과 기록 양식

```md
## Smoke test result

- 환경: production / preview / staging
- 배포 SHA:
- 실행 시각:
- 실행자:
- 결과: pass / conditional pass / fail
- 즉시 중단 이슈:
- 런칭 가능하나 추적할 이슈:
- 확인한 테스트 계정:
- Sentry issue 링크:
- Vercel deployment 링크:
- Supabase migration 기준:
```

조건부 통과는 반드시 이슈 링크와 재확인 기한을 남깁니다. 즉시 중단 이슈가 하나라도 있으면 공개 전환 또는 트래픽 확대를 보류합니다.
